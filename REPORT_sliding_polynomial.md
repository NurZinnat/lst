# Отчёт: осцилляции метода 1 (сдвигающийся многочлен Ньютона)

**Файлы:** `sliding_polynomial.cpp`, `methods_draw.cpp`, `erros.cpp`  
**Тесты:** `test_sliding.cpp` (сборка: `g++ -O2 -std=c++17 -o test_sliding test_sliding.cpp sliding_polynomial.cpp functions.cpp`)  
**Сценарий:** `f(x)=x^3`, `k=3` → `poly_k=5`, отрезок `[-1, 1]`, `n=524388`, ширина окна `W≈1000` пикселей.

---

## Краткий вывод

Осцилляции и «захламление» экрана при больших `n` вызваны **не ошибкой в формулах Ньютона и не потерей точности разделённых разностей на узлах**, а **ошибкой синхронизации скользящего окна с отрисовкой**: за один шаг по экрану выполняется **не более одного** сдвига окна, тогда как при `n ≫ W` за шаг нужно сдвигать окно **много раз** (примерно `n/W` узлов).

Ядро `sliding_polynomial` при **правильном** окне даёт машинную точность на `x^3` даже при `n=524388`.

---

## 1. Как устроен метод в коде

1. На сетке из `n` интервалов шаг `h = (b-a)/n`.
2. Окно из `k` подряд идущих узлов; коэффициенты — разделённые разности в локальных координатах `points[i] = i·h` (`calc_current_coeffs`).
3. Значение — схема Горнера по форме Ньютона (`eval`).
4. При отрисовке справа налево: старт с правого окна (`calc_rightmost_coeffs`), при переходе влево — `calc_next_coeffs` (`first_point_index--`).

---

## 2. Где баг

### 2.1. Основной дефект — `methods_draw.cpp` и `erros.cpp`

```24:26:methods_draw.cpp
      if (i > 0 && x_curr <= sp.get_current_left ())
        sp.calc_next_coeffs ();
```

За одну колонку пикселя (`hx = (b-a)/W`) выполняется **ровно один** сдвиг, хотя смещение по `x` равно `hx`, а шаг сетки `h = (b-a)/n`.

Когда **`hx > h`**, т.е. **`n > W`**, за один пиксель мы «перепрыгиваем» через несколько положений окна. Окно отстаёт, затем `first_point_index` уходит в **отрицательные** значения (ограничения нет), в `calc_current_coeffs` берутся узлы **вне** `[a,b]`, интерполянт превращается в неустойчивую экстраполяцию → огромные значения и визуальные осцилляции.

**Порог (тесты, `W=1000`):**

| n     | hx/h ≈ n/W | max ошибка (draw) | max ошибка (правильные сдвиги) |
|-------|------------|-------------------|--------------------------------|
| 10000 | 10         | 0.41              | ~1e-16                         |
| 12800 | 12.8       | **1.68**          | ~1e-16                         |
| 524388| 524        | **7.5·10⁶**       | ~1e-16                         |

При **`W = n`** (один пиксель на узел) `err_draw` остаётся ~10⁻¹⁵ даже при `n=524388` — алгоритм сдвига сам по себе согласован с сеткой.

### 2.2. Сопутствующий дефект — нет защиты индекса окна

```40:47:sliding_polynomial.cpp
void
sliding_polynomial::calc_next_coeffs ()
{
  first_point_index--;
  calc_current_coeffs ();
}
```

При отставании отрисовки `first_point_index` становится отрицательным; `get_x(global_idx)` уходит левее `a`, значения `f(x)` берутся не с рабочей сетки.

### 2.3. Что **не** является причиной (для `x^3`, `k=5`)

Тест `err_nodes`: для каждого узла сетки окно выставляется корректно, `|P(x_i)-f(x_i)|` при всех `n` из таблицы ≤ **4·10⁻¹⁶**.

Разделённые разности в `calc_current_coeffs` на **равномерных узлах окна** при правильном `first_point_index` устойчивы.

## 3. Когда начинают «скакать» точки на графике

Критерий в тестах: соседние значения на соседних пикселях отличаются более чем на `0.5` (`count_jumps`).

| n (W=1000) | число скачков | max |y_draw| |
|------------|---------------|----------|
| 10000      | 14            | 0.41     |
| 20000      | 333           | 11       |
| 50000      | 784           | 512      |
| 524388     | **975**       | **7.5·10⁶** |

Скачки появляются сразу после того, как `n` превышает ширину окна в пикселях; их число растёт вместе с `n/W`. На левой части отрезка (`x≈-1`) ошибка максимальна — окно «застряло» с большим `first_point_index`, пока индекс не ушёл в минус.

---

## 4. Рекомендуемое исправление

**Вариант A (минимальный)** — в `draw_method1` / `draw_method1_error` / `calc_method1_error`:

```cpp
while (i > 0 && x_curr <= sp.get_current_left ())
  sp.calc_next_coeffs ();
```

**Вариант B** — не сдвигать по пикселям, а вычислять `first_point_index` из `x` (целочисленно через индекс узла) и пересчитывать коэффициенты только при смене окна (дороже по пересчёту, проще логика).

**Вариант C** — в `calc_next_coeffs`:

```cpp
if (first_point_index > 0)
  first_point_index--;
```

Это не заменяет цикл сдвигов, но не даст уходить индексу в минус.

## 5. Как воспроизвести тесты

```bash
g++ -O2 -std=c++17 -o test_sliding test_sliding.cpp sliding_polynomial.cpp functions.cpp

# Режим как в GUI (W=1000) — ломается при больших n
./test_sliding 3 1000

# W=n — не ломается
./test_sliding 3 524388
```

Столбцы вывода:

- `err_nodes` — точность на узлах сетки (эталон ядра);
- `err_draw` — как в `methods_draw.cpp` (один сдвиг на пиксель);
- `err_correct` — справа налево с `while` до попадания `x` в окно;
- `jumps_d` / `jumps_c` — число резких скачков на полилинии.

## 6. Связь с требованиями практикума

- Подпрограммы построения/вычисления в отдельном файле — выполнено.
- Для больших `n` (до 10⁷) кусочный метод должен быть быстрым; для метода 1 при отрисовке важно **не привязывать число сдвигов к числу пикселей**, а к положению `x` на сетке из `n` точек.
- Проверка точности на многочлене степени `k-1` (для `poly_k=5` на `x^3`) после исправления отрисовки должна снова давать порядок машинной ε на всём отрезке.
